# Feature Specification: 统一服务层角色码解析入口

**Feature Branch**: `agent/zcode/co373-unify-rolecode`

**Created**: 2026-06-27

**Status**: Draft

**Input**: User description: "所有服务层权限校验统一走 OSS 缓存 roleCode，彻底废弃直接调 User.getRoleCode()。这是个系统性问题，不是单点 patch。"

**关联缺陷**: Linear CO-373「[BUG] 标书制作阶段投标负责人/辅助人员无权提交标书审核和提交投标」

## 背景与根因（已确认）

OSS（组织系统）用户登录时，系统从 OSS 实时抓取角色并写入本地权限缓存（8 小时 TTL），缓存中存的是正确的内部角色码（如 `bid-Team`、`bid-projectLeader`）。

但本地用户实体的角色码字段（`role_id`）对 OSS 同步用户为 `NULL`，导致实体回退方法返回硬编码的 `"manager"`。

服务层权限校验存在两套分歧的读取路径：
- 4 处服务走 OSS 缓存（正确）
- 19 处服务直调实体回退方法（错误，读到 `"manager"`）

当 OSS 用户（如投标负责人王占俊）被分配任务、提交标书审核、提交投标时，服务层读到 `"manager"`，不在角色白名单内，被统一拒绝（403）。前两次修复（PR #1226、#1241）只补了单点，未统一入口，故反复复发。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 投标负责人可分配任务并管理任务看板 (Priority: P1)

作为投标负责人，我登录系统后被分配为某项目的投标负责人，当我进入项目想给团队成员分配任务、修改任务执行人时，系统不应报"无权管理任务"错误，因为我的真实角色是投标团队成员/负责人，有权管理本项目任务。

**Why this priority**: 这是 CO-373 报障的核心阻断点——投标负责人完全无法使用任务看板，直接阻塞标书制作流程。根因已定位为任务权限守卫直调实体回退方法读到 `"manager"`。

**Independent Test**: 以一个 `role_id=NULL` 的 OSS 用户身份，调用任务分配接口，应成功返回而非 403。

**Acceptance Scenarios**:

1. **Given** 一个 OSS 同步用户（`role_id=NULL`，OSS 缓存角色为 `bid-Team`）已登录并是项目投标负责人，**When** 该用户调用任务分配接口给团队成员派活，**Then** 接口返回成功，任务被正确分配
2. **Given** 同一用户进入任务看板，**When** 查看本项目任务列表，**Then** 能看到全部本项目任务，不因角色被误判为 `"manager"` 而被数据范围过滤掉
3. **Given** 该用户尝试修改任务执行人，**When** 提交修改，**Then** 修改成功，不报 403

---

### User Story 2 - 投标负责人/辅助人员可提交标书审核与提交投标 (Priority: P1)

作为投标负责人或投标辅助人员，当标书编制完成、所有任务已完成且标书文件已上传时，我应能提交标书审核；审核通过后应能提交投标。不应因角色被误判为 `"manager"` 而被拦截。

**Why this priority**: 与 US1 同为 CO-373 的核心阻断点，且是前两次修复尝试覆盖但仍复发的点。根因是标书提交授权策略读到了 `"manager"`。

**Independent Test**: 以 `role_id=NULL`、OSS 缓存角色为 `bid-Team` 的用户身份调用标书审核提交接口与投标提交接口，应成功。

**Acceptance Scenarios**:

1. **Given** 一个 OSS 用户（缓存角色 `bid-Team`）是项目投标负责人，且项目所有任务已完成、标书文件已上传，**When** 该用户提交标书审核，**Then** 审核流程启动成功，不报 403
2. **Given** 标书审核已通过，**When** 该用户提交投标，**Then** 投标提交成功，不报 403
3. **Given** 一个本地建号用户（`role_id` 非空，角色为 `bid-projectLeader`），**When** 该用户提交标书审核，**Then** 行为不变（回归保护：本地用户不受影响）

---

### User Story 3 - 标书审核人搜索框在投标负责人登录时正常显示 (Priority: P2)

作为投标负责人，我登录后进入标书制作阶段，标书审核人模糊搜索框应正常显示并可使用，不应因前端拿到误判的角色码 `"manager"` 而隐藏该控件。

**Why this priority**: CO-373 报障的连锁症状，根因同上。前端按角色码控制控件可见性，拿到 `"manager"` 时控件被隐藏。

**Independent Test**: 以投标负责人身份登录进入标书制作页，标书审核人搜索框可见且可用。

**Acceptance Scenarios**:

1. **Given** 投标负责人（OSS 缓存角色 `bid-projectLeader` 或 `bid-Team`）已登录，**When** 进入标书制作阶段页面，**Then** 标书审核人搜索框正常显示
2. **Given** 该用户在搜索框输入姓名关键字，**When** 触发搜索，**Then** 返回候选人列表，可选择审核人

---

### User Story 4 - 投标辅助人员分配后回显不丢失 (Priority: P2)

作为项目管理员，我给项目分配了投标辅助人员，保存后进入起草阶段，再返回立项阶段时，投标辅助人员字段应显示已分配的人员姓名，不应显示"未分配"。

**Why this priority**: CO-373 报障的问题1。根因是前端回显逻辑对辅助人员字段做了 truthy 判断但无兜底，与负责人字段处理不一致。数据库中辅助人员 ID 实际存在，未丢失。

**Independent Test**: 分配辅助人员 → 保存 → 切换到起草阶段 → 返回立项阶段，辅助人员字段应显示已分配人员姓名。

**Acceptance Scenarios**:

1. **Given** 项目已分配投标辅助人员（数据库 `secondary_lead_user_id` 有值），**When** 用户从起草阶段返回立项阶段查看表单，**Then** 投标辅助人员字段显示已分配人员姓名，不显示"未分配"
2. **Given** 项目未分配投标辅助人员，**When** 查看立项表单，**Then** 显示"未分配"（正常空状态）

---

### Edge Cases

- **OSS 缓存未命中（miss）**：用户是 OSS 用户但缓存已过期或未写入（如后端重启后旧 JWT 仍有效）。系统应 fail-closed（拒绝带敏感操作）并记录告警日志，不能误用实体的 `"manager"` 回退值放行。这与现有 `resolveEffectiveRoleCode` 的行为一致。
- **本地建号用户（非 OSS）**：`external_org_source_app` 为空，应直接用实体角色码（`role_id` 非空），行为不变。
- **缓存角色码为空字符串**：应视为未命中，走 OSS 用户的 fail-closed 路径。
- **并发角色变更**：用户在登录会话中途角色被管理员调整——本期不处理实时刷新，沿用 8 小时缓存 TTL 现状，仅保证读取入口统一。
- **管理员（admin）本地用户**：`role_id` 指向 admin 角色档案，实体回退方法返回正确值，不受本次改动影响。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供唯一的"有效角色码"解析入口，所有服务层权限校验统一调用该入口，不得绕过它直调实体回退方法。
- **FR-002**: 有效角色码解析入口必须按以下优先级取值：(1) OSS 权限缓存命中 → 返回缓存角色码；(2) 非 OSS 用户（无外部组织来源标识）→ 返回实体角色码；(3) OSS 用户但缓存未命中 → 返回空值并记录告警（fail-closed），不得回退为 `"manager"`。
- **FR-003**: 现有直调实体回退方法的 19 处服务层落点（任务权限守卫、标段命令守卫、项目任务工作流守卫、项目可见范围服务、批量分单策略、候选人过滤策略、设置服务、平台账号服务、角色档案服务、当前用户解析器等）必须全部改为调用统一入口。
- **FR-004**: 统一入口的纯核心解析逻辑必须可单元测试，不依赖框架（遵循 FP-Java Profile：纯核心不注入缓存组件，缓存读取由外壳层完成后传入）。
- **FR-005**: 现有已走 OSS 缓存的 4 处服务（标书编制服务、数据范围服务、用户详情服务、认证服务）应改为复用统一入口，消除重复实现。
- **FR-006**: 前端投标辅助人员字段回显必须与投标负责人字段采用一致的兜底策略：当接口返回的人员姓名为空但人员 ID 存在时，应触发姓名补查或显示占位文本，不得直接显示"未分配"。
- **FR-007**: 本地建号用户（非 OSS）的角色码解析行为必须保持不变（回归保护）。
- **FR-008**: 管理员（admin）用户的角色码解析行为必须保持不变（回归保护）。
- **FR-009**: 统一入口必须记录解析决策日志（命中缓存 / 本地用户 / 缓存未命中 fail-closed），便于线上排查权限问题。
- **FR-010**: 实体回退方法本身不修改（保持可单测性），但不得再被服务层权限校验直接调用；它仅作为统一入口的内部回退之一被调用。

### Assumptions

- OSS 权限缓存的 8 小时 TTL 与 Redis/内存降级策略不变。
- 实体 `role_id=NULL` 对 OSS 用户的现状不变（不在本次修复数据）。
- 前端角色码来源仍由登录/会话接口返回，本次只修后端解析入口；前端拿到正确角色码后控件可见性自然恢复（US3）。
- 本期不引入缓存主动刷新机制，沿用登录时写入、登出时清除的现状。

## Success Criteria

1. **权限阻断消除**：OSS 用户（缓存角色 `bid-Team` / `bid-projectLeader`）以投标负责人/辅助人员身份执行任务分配、标书审核提交、投标提交时，100% 不再出现 403 权限拒绝。
2. **读取入口统一**：服务层权限校验路径中直调实体回退方法的生产代码落点数降为 0（由代码搜索验证）；统一入口成为唯一调用点。
3. **回归零退化**：本地建号用户与管理员的所有既有权限行为保持不变，相关既有测试全部通过。
4. **可验证性**：新增纯核心解析逻辑有独立单元测试覆盖三种取值路径（缓存命中 / 本地用户 / 缓存未命中）；各 Guard/Service 落点有测试验证调用统一入口。
5. **可观测性**：线上权限问题排查时，可通过日志（traceId + 解析决策）确认某请求实际使用的角色码及其来源，无需再 SSH 翻日志推测。
6. **前端回显**：投标辅助人员字段在数据存在时 100% 正确回显，不再误显示"未分配"。

## Key Entities

- **OSS 权限缓存条目**：用户名 → {角色码, 菜单权限, 原始权限, 过期时间}，8 小时 TTL，Redis 优先内存降级。
- **用户实体**：含 `role_id`（可空，指向角色档案）、`external_org_source_app`（OSS 来源标识，非空表示 OSS 用户）。
- **角色档案**：`code`（内部角色码，如 `bid-Team`）、`name`（显示名）。
- **有效角色码**：本次新增的解析结果，由统一入口产出，是服务层权限校验的唯一输入。
