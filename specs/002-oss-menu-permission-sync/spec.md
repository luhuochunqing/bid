# Feature Specification: OSS Menu Permission Sync

**Feature Branch**: `002-oss-menu-permission-sync`

**Created**: 2026-06-18

**Status**: Draft

**Input**: User description: "接入 OSS 用户菜单树接口 GET /sysMenuUrl/getUserMenuTree，按角色同步/配置西域数智化投标管理平台内部角色的菜单权限。支持在系统设置中按角色触发同步，也支持在组织架构同步流程中可选自动聚合。要求真实 API、批量/调用可观测、大小写安全映射、FP-Java 架构合规。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理员按角色从 OSS 同步菜单权限 (Priority: P1)

作为系统管理员，我希望在角色管理页面选择一个内部角色，并指定一个具有该角色代表性的 OSS 用户（工号），一键从 OSS 拉取该用户在 OSS 侧的菜单树，自动映射为平台内部的菜单权限码，写入该角色，从而减少人工逐条勾选菜单的重复工作。

**Why this priority**: 直接满足用户提出的“通过 OSS 接口配置不同角色在我们系统的菜单权限”的核心诉求；管理员可以基于 OSS 的真实权限配置快速初始化/校正平台角色权限。

**Independent Test**: 可以在系统设置的角色管理页面完成一次“从 OSS 同步菜单权限”操作，操作后该角色的菜单权限与 OSS 返回的菜单编码集合一致。

**Acceptance Scenarios**:

1. **Given** 管理员在角色管理页面选中角色“投标项目负责人”并输入代表工号，**When** 点击“从 OSS 同步菜单权限”，**Then** 系统调用 OSS 菜单树接口，返回成功后该角色的菜单权限被更新为 OSS 菜单树映射后的内部权限码集合，并提示同步成功。
2. **Given** 同一个角色再次执行同步且 OSS 返回的菜单集合发生变化，**When** 同步完成，**Then** 系统以最新 OSS 菜单集合覆盖该角色的旧菜单权限，保留角色其他属性不变。
3. **Given** OSS 接口返回为空菜单树，**When** 同步完成，**Then** 系统清空该角色的菜单权限（或保持空集合），并给出明确提示，不报错。

---

### User Story 2 - 组织架构同步时自动按用户聚合角色菜单权限 (Priority: P2)

作为平台运营人员，我希望在每日/定时的组织架构同步过程中，可选地自动拉取每个已同步 OSS 用户的菜单树，并按其解析出的内部角色聚合菜单权限，使角色权限随着 OSS 配置变化自动演进。

**Why this priority**: 降低长期维护成本，避免每次 OSS 菜单变更都需要管理员手动同步；与现有组织架构同步流程自然衔接。

**Independent Test**: 开启自动同步开关后，执行一次组织架构同步，能够观察到相关角色的菜单权限根据 OSS 用户的菜单树发生变化。

**Acceptance Scenarios**:

1. **Given** 自动同步开关开启，**When** 组织架构同步完成一批用户，**Then** 系统按这批用户解析出的角色聚合 OSS 菜单权限，并合并更新到对应角色的菜单权限字段。
2. **Given** 自动同步开关关闭，**When** 组织架构同步完成，**Then** 系统不调用 OSS 菜单树接口，角色菜单权限保持不变。
3. **Given** 某个用户拉取 OSS 菜单树失败，**When** 同步流程结束，**Then** 失败仅影响该用户，其他用户和角色的聚合结果正常写入，并记录失败日志。

---

### User Story 3 - 映射关系可配置且大小写安全 (Priority: P3)

作为系统管理员/运维，我希望 OSS 菜单编码到平台内部权限码的映射规则可以配置，并且对 OSS 侧的大小写、空格、前缀差异具有容错能力，避免因为 OSS 配置细节差异导致权限丢失。

**Why this priority**: 提升集成的健壮性，符合 Constitution 中“角色映射大小写安全”的非协商原则。

**Independent Test**: 修改配置中的映射规则后，同步同一组 OSS 菜单，角色的菜单权限按新规则更新。

**Acceptance Scenarios**:

1. **Given** 配置中定义 OSS 菜单编码“ProjectManager”映射到内部权限码“project.manager”，**When** OSS 返回“projectmanager”或“PROJECTMANAGER”，**Then** 系统仍能正确映射到“project.manager”。
2. **Given** 配置中未定义某个 OSS 菜单编码的映射，**When** 同步完成，**Then** 该菜单编码被忽略或按默认规则（如直接使用编码）处理，并在日志中标记未映射项。

---

### Edge Cases

- OSS 接口超时或返回非 200 状态码时，手动同步应给出友好错误提示；自动同步应跳过失败用户并继续。
- OSS 返回的菜单树包含循环父子关系或缺少 parentId 时，系统应能安全解析，不因此崩溃。
- 目标角色在平台中不存在时，手动同步应提示角色不存在；自动同步应跳过该用户并记录。
- 多个用户映射到同一角色时，角色菜单权限应取这些用户菜单权限的并集。
- 同步过程中角色被删除或并发修改时，系统应保证数据一致性（最后一次成功写入生效，或以乐观锁方式处理）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 能够通过 OSS 接口 `GET /sysMenuUrl/getUserMenuTree` 拉取指定用户在指定系统下的菜单树，并解析为结构化的菜单节点列表。
- **FR-002**: 系统 MUST 支持在系统设置的角色管理页面为任意内部角色触发一次 OSS 菜单权限同步，同步时需要指定一个 OSS 用户标识（工号或用户 ID）。
- **FR-003**: 系统 MUST 支持将 OSS 菜单树中的每个节点映射为平台内部的菜单权限码；映射规则可配置，默认对 OSS 菜单编码进行大小写不敏感、去除前后空格的规范化处理。
- **FR-004**: 系统 MUST 在手动同步完成后更新目标角色的菜单权限字段，覆盖旧值，并保留角色的其他属性不变。
- **FR-005**: 系统 MUST 在组织架构同步流程中提供可开关的自动聚合能力：开启后，对每个已同步用户拉取其 OSS 菜单树，按解析出的内部角色取菜单权限并集，写入对应角色。
- **FR-006**: 系统 MUST 记录每次 OSS 菜单树调用的请求/响应摘要、耗时、错误信息，便于排查。
- **FR-007**: 系统 MUST 在 OSS 菜单树接口失败时，手动同步返回明确错误；自动同步跳过失败用户，不影响同步主流程。
- **FR-008**: 系统 MUST 在 OSS 返回菜单编码无法映射到内部权限码时，提供可配置的默认行为：忽略该编码，或使用规范化后的编码作为内部权限码，并记录未映射项。
- **FR-009**: 系统 SHOULD 支持按角色查看最近一次 OSS 菜单同步时间、同步来源（手动/自动）及同步结果摘要。

### Key Entities *(include if feature involves data)*

- **RoleProfile**: 平台内部角色，包含菜单权限字段；是同步的目标对象。
- **OSS User Reference**: 用于向 OSS 标识一个用户的工号或用户 ID，作为菜单树查询的上下文。
- **OSS Menu Tree Node**: OSS 返回的菜单树节点，包含菜单编码、名称、类型、父节点、请求地址、路由路径等属性。
- **Internal Permission Key**: 平台内部用于鉴权与前端展示的菜单权限码，通常对应 RoleProfile.menuPermissions 中的一项。
- **Menu Mapping Rule**: 从 OSS 菜单编码到内部权限码的映射规则，支持大小写不敏感匹配。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员可以在 3 次点击内完成一次角色 OSS 菜单权限同步，同步结果在 5 秒内返回（不含 OSS 接口自身耗时）。
- **SC-002**: 开启自动聚合后，组织架构同步流程对单个用户的额外菜单树查询耗时不超过 1 秒（P95）。
- **SC-003**: OSS 菜单编码到内部权限码的映射准确率不低于 95%（以已配置映射规则的编码为基准）。
- **SC-004**: 因 OSS 接口失败导致的同步中断次数为 0；失败用户/角色被隔离处理，不影响其他数据。
- **SC-005**: 所有角色菜单权限变更均可通过审计日志或同步记录追溯来源（手动/自动）、时间和 OSS 用户标识。

## Assumptions

- OSS 菜单树接口的鉴权方式与现有 OSS 组织架构接口一致，可通过统一凭证和消息头发起请求。
- 平台内部菜单权限码的命名规范在 `RoleProfileCatalog` 中已有定义，映射规则可基于该目录维护。
- 每个内部角色在 OSS 侧至少可以找到一个代表性用户；无法找到时由管理员手动指定。
- 自动聚合功能默认关闭，由运维人员在配置中显式开启，以避免大规模同步时对 OSS 造成突发流量。
- OSS 返回的菜单树深度不超过 5 层，节点数量不超过 1,000 个；超过时可按需求再次拆分优化。
- 同步以 OSS 侧菜单编码为权威来源，平台侧角色菜单权限会被覆盖；需要保留自定义权限的角色可通过配置排除。
